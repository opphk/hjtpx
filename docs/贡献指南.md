## 行为准则

作为贡献者，请您遵守以下行为准则：

1. **保持友善**：与社区成员友好交流，尊重他人观点
2. **包容多样**：欢迎不同背景和经验水平的贡献者
3. **专注专业**：技术讨论应保持专业，避免人身攻击
4. **开放透明**：公开讨论决策过程，接受建设性批评
5. **承担责任**：对代码质量负责，确保提交内容安全可靠

## 开始贡献

### Fork 项目

1. 访问 GitHub 仓库：https://github.com/opphk/behavior-verification-system
2. 点击右上角的 **Fork** 按钮创建您自己的副本

### 克隆您的 Fork

```bash
git clone https://github.com/YOUR_USERNAME/behavior-verification-system.git
cd behavior-verification-system
```

### 添加上游仓库

```bash
git remote add upstream https://github.com/opphk/behavior-verification-system.git
```

### 保持同步

```bash
# 获取上游最新代码
git fetch upstream

# 切换到 main 分支
git checkout main

# 合并上游代码
git merge upstream/main

# 推送更新到您的 Fork
git push origin main
```

## 开发环境

### 环境要求

- **Go**: 1.25+
- **Git**: 2.30+
- **PostgreSQL**: 12+ (开发用)
- **Redis**: 6+ (开发用)
- **Docker**: 20.10+ (可选)

### IDE 配置

推荐使用 VS Code 或 GoLand，并安装以下扩展：

- **VS Code**：
  - Go 扩展
  - ESLint (前端代码)
  - Prettier (代码格式化)

- **GoLand**：
  - 已内置 Go 支持
  - 安装 Go 相关插件

### 本地运行

```bash
# 1. 克隆并进入项目
git clone https://github.com/opphk/behavior-verification-system.git
cd behavior-verification-system

# 2. 启动依赖服务（使用 Docker）
docker-compose up -d postgres redis

# 3. 安装 Go 依赖
cd backend
go mod download

# 4. 配置环境变量
export POSTGRES_HOST=localhost
export POSTGRES_PORT=5432
export REDIS_HOST=localhost
export REDIS_PORT=6379
export JWT_SECRET=your-dev-secret-key

# 5. 运行测试
go test ./...

# 6. 启动服务
go run ./cmd/api/main.go
```

### 测试覆盖

运行单元测试：

```bash
cd backend
go test -v -cover ./...
```

运行特定包测试：

```bash
go test -v ./internal/api/handler/...
go test -v ./internal/service/...
```

## 代码规范

### Go 代码规范

1. **格式化**
   - 使用 `gofmt` 或 `go fmt` 格式化代码
   - 推荐使用 `goimports` 自动管理 import

2. **命名规范**
   ```go
   // 变量命名：使用驼峰命名
   var userName string
   var maxAttempts int

   // 常量命名：使用全大写下划线分隔
   const MaxRetryAttempts = 3
   const DefaultTimeout = 30

   // 函数命名：使用驼峰命名，公开函数首字母大写
   func GetUserByID(id int) (*User, error)
   func (u *User) Validate() error

   // 类型命名：使用驼峰命名，公开类型首字母大写
   type UserService struct{}
   type VerificationResult struct{}
   ```

3. **注释规范**
   ```go
   // PackageName 提供简短描述
   //
   // 详细说明（可选）
   //
   // 示例：
   //   result := PackageName.Function()
   package auth

   // ValidateToken 验证JWT Token的有效性
   //
   // 参数：
   //   - token: JWT Token字符串
   //
   // 返回：
   //   - claims: Token解析后的Claims
   //   - error: 验证失败时的错误
   func ValidateToken(token string) (*Claims, error) {
       // 实现
   }
   ```

4. **错误处理**
   ```go
   // 错误包装
   if err != nil {
       return fmt.Errorf("failed to get user: %w", err)
   }

   // 哨兵错误
   var ErrUserNotFound = errors.New("user not found")
   
   // 错误检查
   if errors.Is(err, ErrUserNotFound) {
       // 处理错误
   }
   ```

5. **包导入顺序**
   ```go
   import (
       // 标准库
       "encoding/json"
       "fmt"
       "io"
       "time"

       // 第三方库
       "github.com/gin-gonic/gin"
       "github.com/golang-jwt/jwt/v5"

       // 项目内部包
       "github.com/opphk/hjtpx/internal/models"
       "github.com/opphk/hjtpx/pkg/response"
   )
   ```

### 前端代码规范

1. **HTML**
   - 使用语义化标签
   - 属性值使用双引号
   - 保持缩进一致（2空格）

2. **JavaScript**
   - 使用 ES6+ 语法
   - 使用 `const` 和 `let`，避免使用 `var`
   - 使用箭头函数
   - 使用模板字符串

   ```javascript
   // 推荐
   const apiClient = {
       async fetchData(url) {
           const response = await fetch(url);
           return response.json();
       }
   };

   // 避免
   var apiClient = {
       fetchData: function(url) {
           return fetch(url).then(function(res) {
               return res.json();
           });
       }
   };
   ```

3. **CSS (Bootstrap)**
   - 优先使用 Bootstrap 提供的类
   - 使用 CDN 加载，禁止自定义 CSS 文件
   - 少量内联样式用于布局微调

### 数据库规范

1. **表命名**：使用单数形式，小写下划线分隔
   ```sql
   CREATE TABLE verification_log (
       id SERIAL PRIMARY KEY,
       session_id VARCHAR(64) NOT NULL,
       created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
   );
   ```

2. **字段命名**：小写下划线分隔
   ```sql
   user_id, created_at, updated_at, is_active
   ```

3. **索引**：为常用查询字段添加索引
   ```sql
   CREATE INDEX idx_verification_session ON verification_log(session_id);
   CREATE INDEX idx_verification_created ON verification_log(created_at);
   ```

## 分支管理

### 分支命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 功能 | `feat/<描述>` | `feat/add-slider-captcha` |
| 修复 | `fix/<描述>` | `fix/login-timeout-issue` |
| 重构 | `refactor/<描述>` | `refactor/optimize-behavior-analysis` |
| 文档 | `docs/<描述>` | `docs/update-api-docs` |
| 测试 | `test/<描述>` | `test/add-unit-tests` |
| 性能 | `perf/<描述>` | `perf/improve-query-speed` |
| 样式 | `style/<描述>` | `style/format-code` |
| 构建 | `build/<描述>` | `build/update-dockerfile` |

### 分支流程

```
main (稳定分支)
│
├── feat/add-slider-captcha (功能分支)
│   └── 开发完成后合并到 main
│
├── fix/login-bug (修复分支)
│   └── 开发完成后合并到 main
│
└── docs/update-readme (文档分支)
    └── 开发完成后合并到 main
```

### 创建分支

```bash
# 从 main 创建新分支
git checkout main
git pull upstream main
git checkout -b feat/add-new-feature
```

## 提交规范

### 提交格式

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

### 类型说明

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(captcha): add slider captcha support` |
| `fix` | 修复Bug | `fix(auth): resolve token expiration issue` |
| `docs` | 文档更新 | `docs: update API documentation` |
| `style` | 代码格式 | `style: format code with gofmt` |
| `refactor` | 重构 | `refactor(service): extract user validation` |
| `perf` | 性能优化 | `perf: improve query performance` |
| `test` | 测试相关 | `test: add unit tests for auth` |
| `build` | 构建相关 | `build: update Dockerfile` |
| `ci` | CI/CD | `ci: add GitHub Actions workflow` |
| `chore` | 其他 | `chore: update dependencies` |

### 范围说明

使用受影响的主要模块作为 scope：

| Scope | 说明 |
|-------|------|
| captcha | 验证码相关 |
| auth | 认证相关 |
| api | API相关 |
| admin | 管理端相关 |
| frontend | 用户端相关 |
| database | 数据库相关 |
| config | 配置相关 |
| test | 测试相关 |
| docs | 文档相关 |

### 提交示例

```bash
# 功能提交
git commit -m "feat(captcha): add slider verification support

- implement slider image generation
- add verification endpoint
- integrate behavior analysis

Closes #123"

# 修复提交
git commit -m "fix(auth): resolve token refresh issue

The token refresh was failing when using long-lived tokens.
Added proper validation for token expiration.

Fixes #456"

# 文档提交
git commit -m "docs: update API documentation

- add new endpoints
- fix typos
- add usage examples"

# 提交检查清单
# 1. 确保代码格式化
gofmt -w .
goimports -w .

# 2. 确保测试通过
go test ./...

# 3. 检查提交信息
git commit -m "type(scope): description"
```

### 提交频率

- 每完成一个功能点或修复一个bug后提交
- 不要积累大量未提交的更改
- 保持提交粒度适中，避免过大的提交

## 测试要求

### 测试类型

1. **单元测试**
   - 测试独立的函数和方法
   - Mock 外部依赖
   - 覆盖率目标：核心模块 80%+

2. **集成测试**
   - 测试多个组件的交互
   - 使用真实的数据库和缓存
   - 放在 `*_test.go` 文件中

3. **端到端测试**
   - 测试完整的用户流程
   - 通过 API 调用测试
   - 放在 `test/e2e/` 目录

### 测试命名

```go
// 正确示例
func TestValidateToken(t *testing.T) { ... }
func TestUserService_GetUserByID(t *testing.T) { ... }
func TestCaptchaVerify(t *testing.T) { ... }

// 正确使用子测试
func TestMath(t *testing.T) {
    t.Run("Add", func(t *testing.T) { ... })
    t.Run("Subtract", func(t *testing.T) { ... })
}
```

### 测试覆盖率

```bash
# 生成覆盖率报告
go test -coverprofile=coverage.out ./...

# 查看覆盖率详情
go tool cover -func=coverage.out

# 生成HTML报告
go tool cover -html=coverage.out -o coverage.html
```

### Mock 使用

```go
// 使用接口和 Mock
type UserRepository interface {
    FindByID(id int) (*User, error)
}

type MockUserRepository struct {
    mock.Mock
}

func (m *MockUserRepository) FindByID(id int) (*User, error) {
    args := m.Called(id)
    if args.Get(0) == nil {
        return nil, args.Error(1)
    }
    return args.Get(0).(*User), args.Error(1)
}

// 在测试中使用
func TestGetUser(t *testing.T) {
    mockRepo := new(MockUserRepository)
    service := NewUserService(mockRepo)

    mockRepo.On("FindByID", 1).Return(&User{ID: 1}, nil)

    user, err := service.GetUser(1)
    assert.NoError(t, err)
    assert.Equal(t, 1, user.ID)

    mockRepo.AssertExpectations(t)
}
```

## 文档要求

### 代码文档

所有公开的函数、类型和常量都需要添加文档注释：

```go
// UserService 处理用户相关业务逻辑
type UserService struct {
    repo UserRepository
}

// GetUser 根据ID获取用户信息
//
// 参数：
//   - id: 用户ID
//
// 返回：
//   - 用户信息指针
//   - 错误信息
func (s *UserService) GetUser(id int) (*User, error) {
    return s.repo.FindByID(id)
}
```

### API 文档

新增 API 端点需要在文档中更新：

1. 更新 `docs/API接口文档.md`

## Pull Request流程

### 创建 Pull Request

1. **推送分支到您的 Fork**
   ```bash
   git push origin feat/your-feature
   ```

2. **在 GitHub 上创建 PR**
   - 选择目标分支：`main`
   - 选择源分支：`feat/your-feature`
   - 填写标题和描述

3. **PR 模板**

   ```markdown
   ## 变更描述
   简要说明本次变更的内容

   ## 变更类型
   - [ ] 新功能 (feat)
   - [ ] 修复bug (fix)
   - [ ] 文档更新 (docs)
   - [ ] 代码重构 (refactor)
   - [ ] 性能优化 (perf)
   - [ ] 测试 (test)
   - [ ] 其他

   ## 关联Issue
   关联的Issue编号，如：Closes #123

   ## 测试计划
   - [ ] 单元测试通过
   - [ ] 集成测试通过
   - [ ] 手动测试完成

   ## 截图/截图（如适用）
   ```

### Review 流程

1. **自动检查**
   - [ ] CI/CD 测试通过
   - [ ] 代码格式检查通过
   - [ ] 测试覆盖率达标

2. **代码审查**
   - 等待维护者 Review
   - 根据反馈修改代码
   - 回复审查意见

3. **合并**
   - 审查通过后由维护者合并
   - 删除已合并的分支

### PR 标题规范

```
feat(captcha): add slider verification support
fix(auth): resolve token refresh issue
docs: update API documentation
refactor(service): improve code structure
```

## 常见问题

### Q: 如何开始贡献？

**A**: 您可以从以下方面开始：
- 查看标记为 `good first issue` 或 `help wanted` 的 Issue
- 完善或翻译文档
- 修复简单的 Bug
- 添加测试用例

### Q: 提交被拒绝怎么办？

**A**: 
- 仔细阅读审查意见
- 根据反馈修改代码
- 如果有不同意见，可以在评论区讨论

### Q: 可以直接提交到 main 分支吗？

**A**: 不可以。所有变更必须通过 Pull Request，并经过代码审查后才能合并。

### Q: 需要签署 CLA 吗？

**A**: 目前不需要 CLA，但您需要确保：
- 代码是您原创的
- 您有权贡献该代码
- 代码使用 MIT 许可证
