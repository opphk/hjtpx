## 简介

本手册提供 HJTPX 验证系统的完整监控运维指南，包括系统监控、告警配置、备份恢复、升级回滚等内容，帮助运维人员确保系统的稳定运行。

### 运维原则

1. **预防胜于救治**：建立完善的监控和预警机制
2. **自动化优先**：减少手动操作，降低人为错误
3. **文档记录**：所有操作都应记录和归档
4. **定期演练**：定期进行故障恢复演练
5. **持续改进**：不断优化运维流程和工具

## 应用监控

### 业务指标

| 指标 | 说明 | PromQL 查询示例 |
|------|------|----------------|
| QPS | 每秒请求数 | rate(http_requests_total[1m]) |
| 错误率 | 错误请求占比 | rate(http_requests_total{status=~"5.."}[1m]) / rate(http_requests_total[1m]) |
| 响应延迟 | 请求延迟分布 | histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m])) |
| 验证码生成量 | 验证码生成总数 | sum(rate(captcha_generated_total[1h])) |
| 验证成功率 | 验证成功比例 | rate(verification_success_total[5m]) / rate(verification_total[5m]) |

### 自定义指标

```go
// 应用代码中的指标定义
var (
    CaptchaGenerated = prometheus.NewCounterVec(
        prometheus.CounterOpts{
            Name: "captcha_generated_total",
            Help: "Total number of captchas generated",
        },
        []string{"type"},
    )
    
    VerificationDuration = prometheus.NewHistogramVec(
        prometheus.HistogramOpts{
            Name:    "verification_duration_seconds",
            Help:    "Verification duration in seconds",
            Buckets: []float64{0.1, 0.2, 0.5, 1, 2, 5},
        },
        []string{"status"},
    )
)

func init() {
    prometheus.MustRegister(CaptchaGenerated)
    prometheus.MustRegister(VerificationDuration)
}
```

## 告警配置

### 告警规则示例

```yaml
# monitoring/prometheus/alerts.yml
groups:
  - name: hjtpx_alerts
    rules:
      # 高错误率告警
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High error rate detected"
          description: "Error rate is {{ $value }} for service {{ $labels.job }}"
      
      # 高延迟告警
      - alert: HighLatency
        expr: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) > 0.5
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High latency detected"
          description: "P99 latency is {{ $value }}s"
      
      # 数据库连接池告罄
      - alert: ConnectionPoolExhausted
        expr: go_sql_open_connections{driver="postgres"} / go_sql_open_connections_max{driver="postgres"} > 0.8
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Database connection pool almost exhausted"
          description: "Connection pool usage is {{ $value }}"
      
      # 服务宕机告警
      - alert: ServiceDown
        expr: up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Service is down"
          description: "{{ $labels.job }} has been down for more than 1 minute"
      
      # 磁盘空间告警
      - alert: DiskSpaceLow
        expr: (node_filesystem_avail_bytes{fstype!="tmpfs"} / node_filesystem_size_bytes{fstype!="tmpfs"}) < 0.1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Low disk space"
          description: "Disk usage is above 90%"
      
      # 内存告警
      - alert: MemoryHigh
        expr: (1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage"
          description: "Memory usage is above 90%"
```

## 日志管理

### 日志收集架构

使用 Promtail + Loki + Grafana 日志堆栈：

1. **Promtail** - 收集容器和主机日志
2. **Loki** - 存储和索引日志
3. **Grafana** - 可视化查询日志

### Promtail 配置示例

```yaml
# monitoring/promtail/promtail.yml
server:
  http_listen_port: 9080

positions:
  filename: /tmp/positions.yaml

clients:
  - url: http://loki:3100/loki/api/v1/push

scrape_configs:
  # 容器日志收集
  - job_name: docker
    docker_sd_configs:
      - host: unix:///var/run/docker.sock
    relabel_configs:
      - source_labels: ['__meta_docker_container_name']
        regex: '/(.*)'
        target_label: 'container_name'
      - source_labels: ['__meta_docker_container_label_com_docker_compose_service']
        target_label: 'service'
```

### 日志查询示例

在 Grafana Explore 中使用 LogQL 查询：

```logql
# 查询应用错误日志
{service="app"} |= "error"

# 查询慢请求
{service="app"} |= "slow"

# 统计错误数量
count_over_time({service="app"} |= "error"[1h])

# 按状态码统计
rate({service="nginx"} |~ "status=\"\\d+\"" | regexp "status=\"(?P<status>\\d+)\"" | line_format "{{.status}}"[5m])
```

### 日志轮转配置

```yaml
# docker-compose.yml 中的日志配置
services:
  app:
    logging:
      driver: "json-file"
      options:
        max-size: "50m"
        max-file: "10"
        compress: "true"
```

## 备份恢复

### 数据库备份

#### 自动备份脚本

```bash
#!/bin/bash
# scripts/backup.sh

BACKUP_DIR="/opt/backups"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=30

# 创建备份目录
mkdir -p $BACKUP_DIR

# PostgreSQL 备份
echo "Backing up PostgreSQL..."
docker-compose exec -T postgres pg_dumpall -U postgres | gzip > $BACKUP_DIR/postgres_$DATE.sql.gz

# Redis 备份
echo "Backing up Redis..."
docker-compose exec redis redis-cli bgsave
sleep 5
docker-compose cp redis:/data/dump.rdb $BACKUP_DIR/redis_$DATE.rdb

# 配置文件备份
echo "Backing up configs..."
tar -czf $BACKUP_DIR/config_$DATE.tar.gz ./backend/config ./docker-compose.yml .env

# 清理旧备份
echo "Cleaning up old backups..."
find $BACKUP_DIR -name "*.sql.gz" -mtime +$RETENTION_DAYS -delete
find $BACKUP_DIR -name "*.rdb" -mtime +$RETENTION_DAYS -delete
find $BACKUP_DIR -name "*.tar.gz" -mtime +$RETENTION_DAYS -delete

echo "Backup completed: $DATE"
```

#### 定时备份配置

```bash
# 编辑 crontab
crontab -e

# 每天凌晨 2 点执行备份
0 2 * * * /opt/hjtpx/scripts/backup.sh >> /var/log/hjtpx-backup.log 2>&1
```

### 数据库恢复

```bash
# PostgreSQL 恢复
gunzip < postgres_20240101_020000.sql.gz | docker-compose exec -T postgres psql -U postgres

# Redis 恢复
docker-compose stop redis
docker-compose cp redis_20240101_020000.rdb redis:/data/dump.rdb
docker-compose start redis
```

## 升级回滚

### 标准升级流程

#### 1. 备份

```bash
# 执行完整备份
./scripts/backup.sh

# 记录当前版本
git log -1 --oneline > current_version.txt
```

#### 2. 拉取新代码

```bash
# 拉取最新代码
git pull origin main

# 查看变更
git log --oneline HEAD@{1}..HEAD
```

#### 3. 更新服务

```bash
# 停止服务
docker-compose down

# 重新构建
docker-compose build

# 启动服务
docker-compose up -d

# 查看启动日志
docker-compose logs -f app
```

#### 4. 验证服务

```bash
# 健康检查
curl http://localhost:8080/health

# 检查服务状态
docker-compose ps

# 运行冒烟测试
# ...执行测试脚本
```

### 回滚流程

如果升级出现问题，按以下步骤回滚：

```bash
# 1. 停止服务
docker-compose down

# 2. 回滚代码
git reset --hard HEAD@{1}

# 3. 恢复数据（如果需要）
# gunzip < /opt/backups/postgres_xxx.sql.gz | docker-compose exec -T postgres psql -U postgres

# 4. 重新构建旧版本
docker-compose build

# 5. 启动服务
docker-compose up -d

# 6. 验证回滚
curl http://localhost:8080/health
```

### 蓝绿部署（高级）

对于高可用性要求的环境，可以采用蓝绿部署：

```bash
# 蓝环境（当前运行）
docker-compose -f docker-compose.blue.yml up -d

# 绿环境（新版本）
docker-compose -f docker-compose.green.yml up -d

# 验证绿环境后，切换流量
# 更新 Nginx 配置或 DNS 指向绿环境

# 最后停止蓝环境
docker-compose -f docker-compose.blue.yml down
```

## 容量规划

### 资源使用预估

| 部署规模 | 应用实例 | CPU (核) | 内存 (GB) | 数据库 (GB) | Redis (GB) |
|---------|---------|---------|----------|-----------|-----------|
| 小型 | 1 | 2 | 4 | 20 | 2 |
| 中型 | 3 | 4 | 8 | 100 | 8 |
| 大型 | 6+ | 8+ | 16+ | 500+ | 32+ |

### 监控指标用于容量规划

| 指标 | 扩容建议 |
|------|---------|
| CPU 使用率持续 > 70% | 增加 CPU 核心或应用实例 |
| 内存使用率持续 > 85% | 增加内存 |
| 数据库连接池使用率 > 80% | 增加连接池大小或数据库实例 |
| Redis 内存使用率 > 80% | 增加 Redis 内存或分片 |

### 扩容策略

1. **垂直扩容** - 增加单实例资源（短期方案）
2. **水平扩容** - 增加实例数量（长期方案，推荐）
3. **数据库分片** - 数据量大时采用

## 高可用配置

### 应用层高可用

```yaml
# docker-compose.yml 中的应用扩展配置
services:
  app:
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: '2'
          memory: 2G
      restart_policy:
        condition: on-failure
```

### 数据库高可用

使用流复制配置 PostgreSQL 主从：

```yaml
# docker-compose.postgres-ha.yml
version: '3.8'

services:
  postgres-master:
    image: postgres:16-alpine
    environment:
      POSTGRES_REPLICATION_MODE: master
      POSTGRES_REPLICATION_USER: repluser
      POSTGRES_REPLICATION_PASSWORD: replpass

  postgres-slave:
    image: postgres:16-alpine
    environment:
      POSTGRES_REPLICATION_MODE: slave
      POSTGRES_REPLICATION_USER: repluser
      POSTGRES_REPLICATION_PASSWORD: replpass
      POSTGRES_MASTER_HOST: postgres-master
    depends_on:
      - postgres-master
```

### Redis 高可用

使用 Redis Sentinel 配置：

```yaml
# docker-compose.redis-ha.yml
services:
  redis-master:
    image: redis:7-alpine

  redis-slave1:
    image: redis:7-alpine
    command: redis-server --slaveof redis-master 6379

  redis-slave2:
    image: redis:7-alpine
    command: redis-server --slaveof redis-master 6379

  sentinel1:
    image: redis:7-alpine
    command: redis-sentinel /usr/local/etc/redis/sentinel.conf

  sentinel2:
    image: redis:7-alpine
    command: redis-sentinel /usr/local/etc/redis/sentinel.conf

  sentinel3:
    image: redis:7-alpine
    command: redis-sentinel /usr/local/etc/redis/sentinel.conf
```
