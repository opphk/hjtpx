## 简介

本指南旨在帮助您优化 HJTPX 验证系统的性能，使其在不同的负载条件下都能稳定高效地运行。我们将从多个维度探讨性能优化的方法。

### 性能优化原则

1. **先测量，后优化**：始终通过基准测试确定当前性能状况
2. **优先优化瓶颈**：先优化最影响系统性能的部分
3. **小步快跑**：每次只优化一个方面，验证效果后再继续
4. **平衡性能与成本**：在性能提升和资源投入之间取得平衡
5. **持续监控**：优化后持续监控，确保优化效果持久

### 连接池优化

#### PostgreSQL 连接池

```yaml

  max_open_conns: 100   # 最大打开连接数，通常是 CPU 核心数的 10-20 倍
```

#### 并发处理

```go
// 限制并发请求数
var semaphore = make(chan struct{}, 1000)

func LimitMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        select {
        case semaphore <- struct{}{}:
            defer func() { <-semaphore }()
            next.ServeHTTP(w, r)
        default:
            http.Error(w, "Too many requests", http.StatusTooManyRequests)
        }
    })
}
```

## Redis 优化

### 内存配置

```yaml
# docker-compose.yml 中的配置
redis:
  command: >
    redis-server
    --requirepass ${REDIS_PASSWORD}
    --maxmemory 2gb
    --maxmemory-policy allkeys-lru
    --appendonly yes
    --appendfsync everysec
```

### 内存策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| allkeys-lru | 移除最少使用的键 | 通用缓存场景 |
| volatile-lru | 仅移除设置了过期时间的键 | 需要保留部分数据 |
| allkeys-random | 随机移除键 | 数据重要性相近 |
| volatile-random | 随机移除设置了过期时间的键 | 混合场景 |
| volatile-ttl | 移除即将过期的键 | 明确数据有效期 |

### 持久化优化

```bash
# RDB 持久化配置
save 900 1      # 900秒内至少1个键变化
save 300 10     # 300秒内至少10个键变化
save 60 10000   # 60秒内至少10000个键变化

# AOF 持久化配置
appendonly yes
appendfsync everysec  # 每秒同步，推荐
# appendfsync always   # 每次写入同步（安全但慢）
# appendfsync no       # 不同步（快但不安全）

# AOF 重写配置
no-appendfsync-on-rewrite yes
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

## 缓存策略

### 多级缓存架构

```
应用层缓存 (本地内存)
    ↓
Redis 缓存
    ↓
数据库
```

### 缓存更新策略

#### 1. Cache-Aside 模式

```go
func GetVerificationLog(sessionID string) (*VerificationLog, error) {
    // 先从缓存获取
    key := fmt.Sprintf("verification:%s", sessionID)
    cached, err := redis.Get(ctx, key).Result()
    if err == nil && cached != "" {
        var log VerificationLog
        json.Unmarshal([]byte(cached), &log)
        return &log, nil
    }
    
    // 从数据库获取
    log, err := db.GetVerificationLog(sessionID)
    if err != nil {
        return nil, err
    }
    
    // 写入缓存
    data, _ := json.Marshal(log)
    redis.Set(ctx, key, data, 5*time.Minute)
    
    return log, nil
}
```

#### 2. Write-Through 模式

```go
func CreateVerificationLog(log *VerificationLog) error {
    // 先写入数据库
    err := db.CreateVerificationLog(log)
    if err != nil {
        return err
    }
    
    // 再写入缓存
    key := fmt.Sprintf("verification:%s", log.SessionID)
    data, _ := json.Marshal(log)
    redis.Set(ctx, key, data, 5*time.Minute)
    
    return nil
}
```

### 缓存失效策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 时间过期 | 设置 TTL (Time To Live) | 热点数据 |
| 手动失效 | 数据变更时主动删除 | 更新频率低的数据 |
| LRU 淘汰 | 内存不足时自动淘汰 | 空间有限的缓存 |

### 缓存预热

```go
func WarmUpCache() {

    for i := 0; i < 100; i++ {
        // 预加载常用数据到缓存
        GenerateAndCacheCaptcha()
    }
    
    log.Info("Cache warm-up complete")
}
```
