# HJTPX 开发核心文档

## 项目概述

### 项目名称
HJTPX - 现代化全栈应用程序

### 项目描述
HJTPX 是一个功能完整的现代化全栈应用程序，提供 RESTful API 接口服务和响应式前端界面。项目采用前后端分离架构，后端使用 Express.js 构建 API 服务，前端使用 React + Vite 构建用户界面。

### 技术栈

#### 后端技术栈
- **框架**: Express.js ^4.18.2
- **数据库**: PostgreSQL ^8.20.0
- **缓存**: Redis ^4.6.0 / ioredis ^5.10.1
- **实时通信**: Socket.io ^4.8.3
- **认证**: JWT (jsonwebtoken ^9.0.3)
- **密码加密**: bcrypt ^5.1.1
- **数据验证**: Joi ^18.2.1
- **限流**: express-rate-limit ^8.5.2
- **安全**: helmet ^8.1.0, cors ^2.8.6
- **日志**: Winston (自定义配置)
- **WebSocket**: ws ^8.20.1

#### 前端技术栈
- **框架**: React ^18.2.0
- **路由**: React Router ^6.20.0
- **构建工具**: Vite ^5.0.0
- **实时通信**: Socket.io-client ^4.8.3
- **状态管理**: React Context API
- **样式**: CSS (全局样式 + 组件样式)
- **代码质量**: ESLint, Prettier

#### 开发工具
- **测试框架**: Jest ^29.7.0
- **代码检查**: ESLint ^8.50.0
- **代码格式化**: Prettier ^3.0.0
- **Git Hooks**: Husky
- **CI/CD**: GitHub Actions
- **代码编辑器**: VS Code

### 项目目标
1. 构建安全、可靠、可扩展的全栈应用程序
2. 提供完整的用户认证和权限管理系统
3. 实现实时数据更新和通知推送功能
4. 提供丰富的数据统计和分析功能
5. 支持移动端访问和 PWA 功能
6. 确保代码质量和测试覆盖率

## 代码规范

### 代码风格指南

#### 通用规范
- 使用 ESLint 进行代码检查
- 使用 Prettier 进行代码格式化
- 文件命名规范：小写字母 + 下划线（如 `user_service.js`）
- 组件命名规范：大驼峰（如 `UserList.jsx`）
- 常量命名规范：大写字母 + 下划线（如 `MAX_LOGIN_ATTEMPTS`）
- 变量命名规范：小驼峰（如 `userName`）
- 函数命名规范：小驼峰（如 `getUserById`）
- 类命名规范：大驼峰（如 `AuthService`）

#### 后端规范（Node.js/Express）
- 使用 `async/await` 处理异步操作
- 统一错误处理（使用自定义错误类）
- 使用参数化查询防止 SQL 注入
- 所有路由必须使用验证中间件
- 敏感配置使用环境变量
- 统一响应格式（`success`、`data`、`error`）

#### 前端规范（React）
- 使用函数组件和 Hooks
- 组件文件使用 `.jsx` 扩展名
- 自定义 Hooks 以 `use` 开头
- Props 类型使用 PropTypes 或 TypeScript
- 优先使用 React Context 进行状态管理
- 组件尽量保持单一职责

### 提交信息规范

使用 Conventional Commits 规范：

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

#### Type 类型
- `feat`：新功能
- `fix`：错误修复
- `docs`：文档更新
- `style`：代码格式（不影响代码运行的变动）
- `refactor`：重构
- `perf`：性能优化
- `test`：测试相关
- `chore`：构建过程或辅助工具的变动

#### 示例
```
feat(auth): add password reset functionality

- implement POST /api/v1/password/forgot
- implement POST /api/v1/password/reset
- add token generation and validation

Closes #123
```

```
fix(auth): resolve JWT token expiration issue

- update token refresh logic
- add token validation error handling

Fixes #456
```

### 代码审查标准

#### 审查清单
1. **功能性**：代码是否实现了预期的功能
2. **代码质量**：是否符合代码规范
3. **安全性**：是否存在安全漏洞
4. **性能**：是否存在性能问题
5. **测试**：是否有适当的测试覆盖
6. **文档**：是否更新了相关文档
7. **错误处理**：是否正确处理了错误情况
8. **可维护性**：代码是否易于理解和维护

#### 审查要求
- 所有 PR 必须通过 CI 检查
- 必须有至少 1 人审批才能合并
- 测试覆盖率必须达标（80%）
- 不能有 ESLint 错误
- 不能有未解决的 console.log（除测试文件）

### 部署流程

#### 自动化部署流程
1. 开发者提交代码到功能分支
2. 创建 Pull Request 触发 CI 检查
3. CI 执行测试、代码检查、构建
4. 代码审查通过后合并到主分支
5. 主分支合并自动部署到预发布环境
6. 预发布环境测试通过后手动部署到生产环境

#### 数据库迁移流程
1. 在 `migrations/` 目录创建迁移文件
2. 迁移文件命名格式：`XXX_description.sql`
3. 执行迁移命令：`npm run migrate`
4. 回滚命令：`npm run migrate:rollback`

#### 回滚流程
1. 检测到问题后立即停止部署
2. 使用 Git 回滚到上一个稳定版本
3. 执行数据库回滚（如需要）
4. 重新运行测试验证
5. 通知相关人员

  - 生产环境：文件 + ELK Stack
- **轮转**：每日轮转，保留 30 天
- **关键字段**：
  - `timestamp`：时间戳
  - `level`：日志级别
  - `message`：日志消息
  - `requestId`：请求 ID
  - `userId`：用户 ID（如适用）
  - `ip`：客户端 IP
  - `method`：HTTP 方法
  - `path`：请求路径
  - `statusCode`：响应状态码
  - `duration`：响应时间
